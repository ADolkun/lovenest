Lovenest fork strategy

Which branch model should we use?

You are maintaining a product fork of securo-finance/securo. The goal is to keep taking upstream fixes while continuing Lovenest-specific work without accidentally mixing the two.

My recommendation

Keep main as the clean Securo mirror, but rename custom to lovenest.

This keeps upstream syncing clean while making the real product branch obvious.

Best compromise

main + lovenest

Preserves a clean upstream mirror and gives your product branch a meaningful name.

Current is okay

main + custom

Technically sound, but custom is vague and harder for contributors to understand.

Modern product default

main = Lovenest

Cleanest for a standalone product, but gives up GitHub fork-sync assumptions.

Options compared

Recommended compromise

Option A: rename custom to lovenest

Best fit if Lovenest is a product fork but you still care about clean upstream syncs.

mainExact mirror of Securo main
lovenestYour real product branch and GitHub default
feature/*Lovenest work branches
upstream-fix/*Branches for PRs to Securo
Why it works
  • main stays clean for upstream syncing.
  • The product branch has a self-explanatory name.
  • Default branch can be lovenest, so visitors see the real project.
  • Upstream PRs can still branch from clean main.
Cost
  • One-time branch rename and settings update.
  • You still use explicit CLI sync, not blindly relying on GitHub's sync button.
  • Need to be careful when opening PRs to upstream.
Acceptable current setup

Option B: keep custom

This is basically what you have now.

mainExact mirror of Securo main
customLovenest product branch
feature/*Lovenest work branches
upstream-fix/*Branches for PRs to Securo
Why it works
  • Very clear technical separation from upstream.
  • Low disruption because it is already configured.
  • Good if the repo remains mostly a fork-maintenance workspace.
Cost
  • custom is not a modern/conventional branch name.
  • New contributors may not understand which branch matters.
  • The repo looks less like a real product project.
Best for standalone product

Option C: make main Lovenest

Most modern if Lovenest is no longer primarily a fork-sync project.

mainLovenest product branch
upstream-mainOptional mirror of Securo main
feature/*Lovenest work branches
upstream-fix/*Based from upstream/main
Why it works
  • Matches modern GitHub convention: default main is the product.
  • Simple for deployment, docs, and contributors.
  • No confusing long-lived custom or dev branch.
Cost
  • Your fork's main no longer mirrors Securo.
  • GitHub's fork sync feature becomes less useful.
  • Upstream PR branches must be created from upstream/main, not product main.
I would avoid

Option D: use dev

This is older Git Flow style. It adds a branch without solving the fork problem.

mainCould be upstream mirror or stable release
devActual work branch
feature/*Work branches
When it makes sense
  • You have formal release trains.
  • You need separate staging and production stabilization branches.
Why not here
  • dev becomes the real product branch anyway.
  • Less modern for small/fast-moving projects.
  • Does not make upstream syncing clearer than lovenest.

The recommended day-to-day workflow

1. Sync Securo into the clean mirror
1
Switch to mirror branch
git switch main
2
Fetch upstream Securo
git fetch upstream
3
Fast-forward only
git merge --ff-only upstream/main
4
Push clean mirror
git push origin main
2. Bring Securo updates into Lovenest
1
Branch from product branch
git switch lovenest
git pull --ff-only origin lovenest
git switch -c sync-main-into-lovenest-YYYYMMDD
2
Merge upstream mirror with ancestry
git merge origin/main
3
Open PR into product branch
gh pr create --repo ADolkun/lovenest --base lovenest
4
Merge with merge commit
Do not squash upstream-sync PRs.

Decision rule

If Lovenest is mainly a product people use, prefer lovenest as the default branch and keep main as the upstream mirror. If Lovenest eventually becomes fully independent, consider making product main later.
